📋一、專案背景

nLLMService 簡介

nLLMService（com.ntx.llmservice）是 Netronix 的 on-device LLM 推論服務，部署於 MediaTek MT8126+ SoC 設備，使用 NPU/APU 硬體加速執行 Qwen 2.5 / Llama 3.2 模型推論。

  • 必須以 System Privileged App 部署（Linker Namespace 限制）
  • 透過 JNI 呼叫 MTK LLM SDK（libmtk_llm.so → libneuron_runtime.so）
  • 開發專案：AIDemoTest1，乾淨版本：nLLMService

BSP 環境（Split Build）

樹路徑Android 版本SELinux 版本號
MSSI (SYS)android16_mtk_system_b0mp5_aiv16202504
VNDandroid16_mtk_vendor_v0mp5_aiv—202404
⚠️ 版本號不一致（202504 vs 202404）是導致 SELinux 問題的核心根因。 Vendor CIL 使用 _202404 後綴，但 System 的 compat 映射文件不含自定義 type。

🔐三、SELinux 解決過程（三階段）

階段 1：Domain 隔離

1
問題

untrusted_app 的 neverallow 阻擋硬體存取。無法直接擴充 untrusted_app 權限。

2
解法：建立專屬 Domain

建立 llm_service_app 獨立 domain，透過憑證綁定：

// 關鍵鏈路
Netronix 簽名憑證 → mac_permissions.xml (seinfo=ntx_app)
    → seapp_contexts (seinfo=ntx_app → domain=llm_service_app)
        → App 啟動時 Zygote 分配 llm_service_app domain

階段 2：硬體存取 + MLS

3
問題：MLS 約束

Multi-Level Security 阻擋跨安全等級的硬體存取。

4
解法
  • llm_service_app 加 mlstrustedsubject（允許跨 MLS 存取）
  • apusys_device 加 mlstrustedobject（允許被跨 MLS 存取）
  • 使用 llm_service_app_data_file 避免 app_data_file 的 neverallow 衝突

階段 3：Compat Mapping 🔑 最終突破口

5
問題：規則被靜默丟棄

階段 1-2 的規則全部正確，但 APU 存取仍然 DENIED。

6
根因：Split Build 版本不一致

Vendor CIL 引用 llm_service_app_202404，但 compat 靜態文件不含此映射 → 規則被靜默丟棄。

7
解法：注入 Compat CIL 映射

在 system/sepolicy/private/compat/202404/202404.cil 追加：

;; Custom type mapping for nLLMService
(typeattribute llm_service_app_202404)
(expandtypeattribute (llm_service_app_202404) true)
(typeattributeset llm_service_app_202404 (llm_service_app))

;; Custom data type mapping
(typeattribute llm_service_app_data_file_202404)
(expandtypeattribute (llm_service_app_data_file_202404) true)
(typeattributeset llm_service_app_data_file_202404 (llm_service_app_data_file))

✅ 最終驗證結果

u:r:llm_service_app:s0:c512,c768  com.ntx.llmservice
getenforce = Enforcing
🎉 完全成功！ nLLMService 成功在 Enforcing 模式下以自定義 llm_service_app domain 運行。

🔬四、關鍵技術發現

A. Compat CIL 分區過濾機制

同一個源文件 202404.cil 被不同 tag 過濾後產生不同輸出：

Tag輸出位置結果
{.plat_private}/system/etc/selinux/mapping/202404.cil✅ 包含
{.system_ext_private}/system_ext/etc/selinux/mapping/202404.cil❌ 不包含
ℹ️ 為什麼仍然生效？ 映射走 plat_private 路徑。init 載入所有分區的 mapping 時會合併，只要任一 mapping 包含即可。

B. expandtypeattribute 驗證陷阱

sepolicy-analyze attribute llm_service_app_202404 永遠顯示 "not defined" — 這是正常的！

❌ 錯誤驗證方式 sepolicy-analyze attribute xxx_202404
→ AOSP 的 expandtypeattribute true 會在編譯時展開並移除所有 _202404 attribute。連 untrusted_app_202404 也查不到。
✅ 正確驗證方式 1. adb shell "grep llm_service /system/etc/selinux/mapping/202404.cil"
2. adb shell "ps -eZ | grep llmservice"
3. adb shell getenforce
4. adb shell dmesg | grep "avc.*llm_service"

C. SELinux 預編譯快取

修改 policy 後若行為不變，可能是舊快取：

adb root
adb shell rm -rf /data/misc/apexdata/com.android.permission/*
adb shell rm -rf /metadata/sepolicy/*
adb reboot

⚠️七、未來必看注意事項

#事項影響
1AOSP 升級時 compat/202404/202404.cil 會被覆蓋需重新加入映射
2兩邊 BSP 的 SELinux 檔案必須同步忘記同步 → build 失敗或 denial
3新增自定義 public type 需一併加 compat mapping否則 vendor allow 規則被靜默丟棄
4sepolicy-analyze attribute xxx_202404 永遠 "not defined"expandtypeattribute 正常行為

📝 新增自定義 Type 的 SOP

未來新增其他 Netronix 自定義 type 時，必須同步執行：

1
宣告 Type — public/xxx.te + private/xxx.te + vendor/xxx.te
2
綁定 Domain — mac_permissions.xml + seapp_contexts
3
加入 Compat Mapping — compat/202404/202404.cil 追加 3 行 CIL
4
雙 BSP 同步 — MSSI 和 VND 的所有檔案保持一致
